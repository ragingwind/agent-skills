주어진 경로의 프로젝트를 읽고, 오랜 경력의 테크 전문 기자 시점에서 한국어 요약 문서를 작성합니다.

## 입력

`$ARGUMENTS`는 분석 대상 프로젝트의 경로입니다.
- 절대/상대 경로 모두 허용
- 비어 있으면 현재 작업 디렉터리(`pwd`)를 사용
- 경로가 존재하지 않으면 사용자에게 확인을 요청하고 중단

## 분석 단계

1. 경로 확인 — `ls`로 디렉터리 구조를 파악합니다.
2. 핵심 문서 우선 읽기 — `README*`, `docs/`, `package.json`, `pyproject.toml`, `Cargo.toml`, `go.mod`, `CHANGELOG*`, `LICENSE` 등을 우선 확인합니다.
3. 소스 구조 파악 — 주요 디렉터리 트리, 진입점 파일, 의존성 목록을 확인합니다.
4. 차별점 추출 — 비슷한 카테고리의 일반적인 프로젝트와 비교했을 때 무엇이 다른지, 무엇을 강조하는지 식별합니다. 근거가 부족하면 해당 항목은 생략합니다.
5. 한국어 요약문 작성 — 아래 작성 원칙을 그대로 따릅니다.

## 작성 원칙 (출력에 반드시 적용)

- 첫 줄에 문서 제목을 적습니다. 다음 단락은 제목 없이 평문 오버뷰로 작성합니다.
- 본문은 불릿 리스트로 요약합니다.
- 장점, 단점, 차별점, 특장점은 같은 토픽끼리 묶어 분류해 작성합니다. 해당 항목의 근거가 본문에 없다면 그 분류 자체를 쓰지 않습니다.
- 마지막 단락은 전체 내용을 아우르는 평문으로 마무리합니다. "마무리", "결론" 같은 제목은 붙이지 않습니다.
- 어려운 용어는 풀어서 설명합니다.
- 분량은 A4 반장 정도로 유지합니다.
- 오랜 경력의 소프트웨어 엔지니어처럼 사고하고, 오랜 테크 전문 기자처럼 씁니다.
- 호소력은 있되 과장하지 않으며, 건조하고 이성적인 톤을 유지합니다.
- 문장은 리듬감을 살리되 과도하게 문학적으로 흐르지 않게 합니다.
- 모든 문장은 존댓말을 사용합니다.
- 각 세부 항목은 `**내용**` 형태의 굵은 글씨 문법을 사용합니다.

## 금지 사항 (네거티브 프롬프트)

- 마크다운 표(`|`) 문법과 구분선(`---`)을 사용하지 않습니다.
- 마크다운 헤딩 문법(`#`, `##`, `###`)을 어디에서도 사용하지 않습니다. 제목, 소제목, 목차, 마무리 슬로건 모두 평문 또는 굵은 글씨로 처리합니다.
- 사실로 확인되지 않은 가설이나 예측은 적지 않습니다.
- 이모지는 사용하지 않습니다.
- 단언적인 사실 적시를 피합니다. "명확합니다"와 같은 확정적 표현은 사용하지 않습니다.
- 장점, 단점, 차별점, 특장점 중 본문에서 근거가 보이지 않는 항목은 통째로 생략합니다. 억지로 채우지 않습니다.
- 오버뷰에 별도의 제목을 달지 않습니다.

## 출력

위 원칙과 금지 사항을 모두 충족하는 한국어 요약문을 사용자에게 평문 응답으로 제시합니다. 별도의 파일로 저장하지 않으며, 사용자가 별도로 요청한 경우에만 파일로 기록합니다.
